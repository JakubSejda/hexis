# Design System SP1 Part 2 — PR 2.5 Implementation Plan (Form Primitives)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `Input` and `Select` primitives, extend `NumberInput` with the new label/hint/error/size props (ADDITIVE — existing callsites keep working), then adopt `Input` on inline `<input className>` sites.

**Architecture:** Input is the canonical text-field primitive — base `<input>` + optional label/hint/error chrome wrapped in a `<label>` container. Select reuses Input's chrome around a native `<select>` with Lucide chevron. NumberInput stays structurally the same (+/- buttons + center input) but gains the Input-family props wired through to the inner `<input>`.

**Tech Stack:** TS 5, React 19, Tailwind v4, Vitest 4 + RTL (jsdom pragma), `lucide-react` (ChevronDown, Search), `cn()`.

**Scope references:**
- Parent spec: `docs/superpowers/specs/2026-04-21-design-system-foundation-part-2-design.md` §3.7 (Input), §3.8 (Select), §5 (NumberInput formalization), §10 PR 2.5
- Previous PR: #4 (merged — display primitives). Main tip: `b623580`.

**"Breaking" interpretation for NumberInput:** Spec §10 PR 2.5 says "breaking, 5 callsites updated". Read literally that implies callsite code changes. Pragmatic read: **the 5 callsites do NOT require code changes** because the new props (label/hint/error/size) are optional. Existing `<NumberInput value onChange step min max suffix />` calls continue to work visually unchanged. Any callsite MAY opt into `label="Weight"` etc. in a FUTURE PR where the label would improve the UI. PR 2.5 ships only additive props + preserves current rendering when they're absent. This interpretation is flagged explicitly here so reviewer can push back if they want literal breaking changes.

**Out of scope:**
- Markdown docs (PR 2.11)
- Storybook (out of Part 2)
- Form-validation library integration (react-hook-form, Zod, etc.) — Input's `error` prop is just string passthrough; consumer wires validation
- Migration of NumberInput callsites to use new label/hint/error (deferred; adopt-as-needed)

Create `src/components/ui/primitive/Input.tsx`:

```tsx
import { forwardRef, useId, type InputHTMLAttributes, type ReactNode } from 'react'
import { Search } from 'lucide-react'
import { cn } from '@/components/ui/utils/cn'

type Variant = 'default' | 'search'
type Size = 'sm' | 'md' | 'lg'

const SIZE_CLASS: Record<Size, string> = {
  sm: 'h-8 text-sm',
  md: 'h-10 text-sm',
  lg: 'h-12 text-base',
}

const BASE =
  'block w-full rounded-md border bg-background text-foreground placeholder:text-muted focus:outline-none focus:ring-2 focus:ring-ring disabled:opacity-50'

type Props = {
  variant?: Variant
  size?: Size
  error?: string
  label?: string
  hint?: string
  iconLeft?: ReactNode
  iconRight?: ReactNode
  className?: string
  id?: string
} & Omit<InputHTMLAttributes<HTMLInputElement>, 'size' | 'className'>

export const Input = forwardRef<HTMLInputElement, Props>(function Input(
  {
    variant = 'default',
    size = 'md',
    error,
    label,
    hint,
    iconLeft,
    iconRight,
    className,
    id,
    ...rest
  }: Props,
  ref
) {
  const autoId = useId()
  const inputId = id ?? autoId

  const effectiveIconLeft =
    variant === 'search' ? (
      <Search data-testid="input-search-icon" size={16} className="text-muted" />
    ) : (
      iconLeft
    )

  const hasIconLeft = Boolean(effectiveIconLeft)
  const hasIconRight = Boolean(iconRight)

  const inputEl = (
    <div className="relative">
      {hasIconLeft ? (
        <span className="text-muted pointer-events-none absolute top-1/2 left-3 -translate-y-1/2">
          {effectiveIconLeft}
        </span>
      ) : null}
      <input
        ref={ref}
        id={inputId}
        aria-invalid={error ? 'true' : undefined}
        className={cn(
          BASE,
          SIZE_CLASS[size],
          hasIconLeft ? 'pl-9' : 'px-3',
          hasIconRight ? 'pr-9' : '',
          error ? 'border-danger' : 'border-border',
          className
        )}
        {...rest}
      />
      {hasIconRight ? (
        <span className="text-muted pointer-events-none absolute top-1/2 right-3 -translate-y-1/2">
          {iconRight}
        </span>
      ) : null}
    </div>
  )

  if (!label && !hint && !error) {
    return inputEl
  }

  return (
    <label htmlFor={inputId} className="flex flex-col gap-1">
      {label ? <span className="text-muted text-xs font-medium">{label}</span> : null}
      {inputEl}
      {error ? (
        <span className="text-danger text-xs">{error}</span>
      ) : hint ? (
        <span className="text-muted text-xs">{hint}</span>
      ) : null}
    </label>
  )
})
```

Key invariants:
- `id` auto-generated via `useId` if not supplied, wired through htmlFor/id for label association.
- `error` sets `aria-invalid="true"` and `border-danger`; hides hint when present (error takes precedence).
- Search variant forces `<Search />` iconLeft; user's iconLeft is overridden.
- Padding adjusts based on icon presence: `pl-9` with iconLeft, `pr-9` with iconRight, `px-3` when neither.
- `Omit<..., 'size' | 'className'>` to avoid collision with native `size` attribute and `className` handled separately.

```tsx
'use client'
import { useId, type ReactNode } from 'react'

type Props = {
  value: number | null
  onChange: (v: number | null) => void
  step?: number
  min?: number
  max?: number
  placeholder?: string
  suffix?: string
  label?: string
  hint?: string
  error?: string
}

export function NumberInput({
  value,
  onChange,
  step = 1,
  min = 0,
  max = 999,
  placeholder,
  suffix,
  label,
  hint,
  error,
}: Props) {
  const id = useId()
  const handle = (delta: number) => {
    const current = value ?? 0
    const next = Math.max(min, Math.min(max, Math.round((current + delta) * 100) / 100))
    onChange(next)
  }

  const core: ReactNode = (
    <div className="inline-flex items-center gap-1">
      <button
        type="button"
        aria-label="snížit"
        className="bg-border text-foreground active:bg-primary h-11 w-11 rounded-lg"
        onClick={() => handle(-step)}
      >
        −
      </button>
      <input
        id={id}
        type="number"
        inputMode="decimal"
        placeholder={placeholder}
        value={value ?? ''}
        onChange={(e) => {
          const v = e.target.value
          onChange(v === '' ? null : Number(v))
        }}
        aria-invalid={error ? 'true' : undefined}
        className={
          'h-11 w-20 rounded-lg border text-center bg-background text-foreground ' +
          (error ? 'border-danger' : 'border-border')
        }
        step={step}
        min={min}
        max={max}
      />
      {suffix ? <span className="text-muted text-xs">{suffix}</span> : null}
      <button
        type="button"
        aria-label="zvýšit"
        className="bg-border text-foreground active:bg-primary h-11 w-11 rounded-lg"
        onClick={() => handle(step)}
      >
        +
      </button>
    </div>
  )

  if (!label && !hint && !error) {
    return core
  }

  return (
    <label htmlFor={id} className="flex flex-col gap-1">
      {label ? <span className="text-muted text-xs font-medium">{label}</span> : null}
      {core}
      {error ? (
        <span className="text-danger text-xs">{error}</span>
      ) : hint ? (
        <span className="text-muted text-xs">{hint}</span>
      ) : null}
    </label>
  )
}
```

Key invariants:
- When no label/hint/error provided, `core` returned directly (same as before — no visual regression for existing callsites).
- When any chrome prop provided, wrapped in `<label>` with label/hint/error rendered.
- `error` → aria-invalid on inner input + border-danger override via string concat (no cn here; NumberInput's className handling is intentionally minimal).

Not touched: `size` prop. The spec §10 lists it but the current button sizing (h-11) doesn't map cleanly to Input's sm/md/lg (h-8/10/12). Adding a size prop would require rethinking button heights and introduce visual change to the 5 existing callsites. Deferring `size` to a future PR keeps THIS PR scope tight and callsites untouched. Flag in PR body.
