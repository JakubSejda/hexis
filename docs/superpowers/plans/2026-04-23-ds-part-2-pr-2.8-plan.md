# Design System SP1 Part 2 — PR 2.8 Implementation Plan (Tabs + Menu + Accordion)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship three Radix-backed compound primitives using the subcomponent-dot pattern (`Tabs.Root/List/Trigger/Content`, `Menu.Root/Trigger/Content/Item/Separator`, `Accordion.Root/Item/Trigger/Content`) with interaction tests, and adopt `Tabs` on `PhotosPageClient` (only organic candidate).

**Architecture:** All three compounds live in `src/components/ui/compound/` as grouped object-exports (`export const Tabs = { Root, List, Trigger, Content }`). Each subcomponent is a thin wrapper over the matching Radix primitive that injects our Tailwind tokens + active-state styles; consumers never reach for `@radix-ui/react-*` directly. The top-level barrel `@/components/ui` already re-exports `./compound`.

**Tech Stack:** TS 5, React 19, `@radix-ui/react-tabs` 1.1, `@radix-ui/react-dropdown-menu` 2.1, `@radix-ui/react-accordion` 1.2 (all three already installed), Vitest 4 + RTL (jsdom pragma), `@testing-library/user-event` for interaction tests, `cn()`.

**Scope references:**
- Parent spec: `docs/superpowers/specs/2026-04-21-design-system-foundation-part-2-design.md` §4 (compound table), §8 (interaction tests list), §10 PR 2.8
- Previous PRs: 2.6 (OPEN, PR #6) and 2.7 (OPEN, PR #7). Main tip at branch-off: `83a3da9`.
- Parallel branches: this PR is independent of #6 and #7. It creates `src/components/ui/compound/` itself because PR #7 hasn't merged yet; one trivial `compound/index.ts` conflict is expected when rebasing against whichever merges first.

**Deviations from parent spec (explicit so reviewer can push back):**
1. **Menu — build, no adoption this PR** — spec §10 PR 2.8 says "Adopt Menu on row-level edit/delete (workout sets, measurements)". The app currently has zero row-level menus: `SetRow` is a tap-to-open-edit-sheet (no ⋮ button); `MeasurementCell` is inline-edit-on-click; workout sessions are `<Link>`-based rows. Adding a Menu on top of these would add a second tap before an action that's currently one tap — a UX regression. Menu ships as an available primitive; adoption happens when a surface actually grows multiple row actions. Same rationale pattern as PR 2.4 (Pill/Tag/Avatar shipped without forced adoption).
2. **Accordion — build, no adoption this PR** — spec §10 PR 2.8 says "Accordion on long Settings sections". `/settings/macros` is 5 toggles, `/settings/plates` is one form, `/settings/export` is one action. No section is "long". Adoption deferred to when Settings actually grows.
3. **Tabs NOT adopted on `/progress`** — spec §10 PR 2.8 says "Adopt Tabs on Progress page (consolidate ad-hoc tabs)". The Progress page is URL-routed (`/progress/body`, `/progress/nutrition`, …) via `ProgressSegmentControl` (`<Link>`-based, full-page loads). Radix Tabs is client-state based — switching would eliminate URL addressability and break shareable deep-links. Leave `ProgressSegmentControl` as-is; Tabs targets genuinely client-state tab UIs.
4. **Tabs IS adopted on `PhotosPageClient`** — 3 client-state view modes (Grid/Timeline/Před×Po) are the one real candidate. Current markup is bespoke `role="tablist"` + `<button>`s; migrate to `Tabs.Root/List/Trigger/Content` with the panel-switch logic moving from inline ternaries to `Tabs.Content`.
5. **`TimeRangePicker` stays bespoke** — uses `role="tab"` but semantically selects a data window, not a content panel. It's a segmented radio, not tabs. Rename debt; skip.
6. **Subcomponent dot-pattern** — spec §4 shows `Dialog.Root`; we ship as `export const Tabs = { Root, List, Trigger, Content }` so consumers write `<Tabs.Root>`. Internal components are defined as named functions plus a closing namespace-like object. No re-shaped default export for these three (subcomposition is the whole point per spec).

**Out of scope:**
- Menu adoption (no existing row-menu surfaces)
- Accordion adoption (no long Settings sections)
- Progress URL-routed SegmentControl migration
- TimeRangePicker rename (future)
- Storybook / markdown docs (PR 2.11)

'use client'
import * as MenuPrimitive from '@radix-ui/react-dropdown-menu'
import type { ComponentProps } from 'react'
import { cn } from '@/components/ui/utils/cn'

function Root(props: ComponentProps<typeof MenuPrimitive.Root>) {
  return <MenuPrimitive.Root {...props} />
}

function Trigger({ className, ...rest }: ComponentProps<typeof MenuPrimitive.Trigger>) {
  return (
    <MenuPrimitive.Trigger
      className={cn('focus-visible:ring-ring focus-visible:ring-2 focus-visible:outline-none', className)}
      {...rest}
    />
  )
}

function Content({ className, sideOffset = 6, ...rest }: ComponentProps<typeof MenuPrimitive.Content>) {
  return (
    <MenuPrimitive.Portal>
      <MenuPrimitive.Content
        sideOffset={sideOffset}
        className={cn(
          'border-border bg-surface-raised text-foreground z-50 min-w-[160px] rounded-md border p-1 shadow-lg',
          className
        )}
        {...rest}
      />
    </MenuPrimitive.Portal>
  )
}

type ItemVariant = 'default' | 'danger'

type ItemProps = Omit<ComponentProps<typeof MenuPrimitive.Item>, 'className'> & {
  variant?: ItemVariant
  className?: string
}

const ITEM_BASE =
  'flex cursor-pointer items-center rounded px-2 py-1.5 text-sm outline-none select-none data-[highlighted]:bg-border data-[disabled]:pointer-events-none data-[disabled]:opacity-40'

const ITEM_VARIANT: Record<ItemVariant, string> = {
  default: 'text-foreground',
  danger: 'text-danger',
}

function Item({ variant = 'default', className, ...rest }: ItemProps) {
  return <MenuPrimitive.Item className={cn(ITEM_BASE, ITEM_VARIANT[variant], className)} {...rest} />
}

function Separator({ className, ...rest }: ComponentProps<typeof MenuPrimitive.Separator>) {
  return (
    <MenuPrimitive.Separator
      className={cn('bg-border my-1 h-px', className)}
      {...rest}
    />
  )
}

export const Menu = { Root, Trigger, Content, Item, Separator }
```

      <Tabs.Root value={view} onValueChange={(v) => setView(v as ViewMode)}>
        <Tabs.List>
          <Tabs.Trigger value="grid">Grid</Tabs.Trigger>
          <Tabs.Trigger value="timeline">Timeline</Tabs.Trigger>
          <Tabs.Trigger value="compare">Před×Po</Tabs.Trigger>
        </Tabs.List>
        {loading ? (
          <p className="text-muted py-8 text-center text-sm">Načítám...</p>
        ) : (
          <>
            <Tabs.Content value="grid">
              <PhotoGrid photos={photos} onPhotoTap={setLightboxIndex} />
            </Tabs.Content>
            <Tabs.Content value="timeline">
              <PhotoTimeline photos={photos} onPhotoTap={setLightboxIndex} />
            </Tabs.Content>
            <Tabs.Content value="compare">
              <BeforeAfter photos={photos} dates={dates} />
            </Tabs.Content>
          </>
        )}
      </Tabs.Root>

      <button
        onClick={() => setUploadOpen(true)}
        className="bg-primary text-background fixed right-4 bottom-20 z-30 flex h-14 w-14 items-center justify-center rounded-full text-2xl font-bold shadow-lg"
        aria-label="Nahrát fotku"
      >
        +
      </button>

      <UploadSheet open={uploadOpen} onOpenChange={onOpenChange => setUploadOpen(onOpenChange)} onUploaded={loadPhotos} />

      {lightboxIndex !== null ? (
        <Lightbox
          photos={photos.map((p) => ({
            id: p.id,
            fullUrl: p.fullUrl,
            takenAt: p.takenAt,
            pose: p.pose,
          }))}
          initialIndex={lightboxIndex}
          onClose={() => setLightboxIndex(null)}
          onDeleted={loadPhotos}
        />
      ) : null}
    </div>
  )
```

Note: preserve the **controlled** pattern (`value={view}` + `onValueChange`) because the component owns `view` state already and other effects may read it. Cast `(v) => setView(v as ViewMode)` because Radix hands us a plain `string`.
